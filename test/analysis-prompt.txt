# GKE Version Upgrade Analysis

## 📊 Version Change Summary

バージョン変更の概要を記載：
- 変更レベル: Major/Minor/Patch
- 変更範囲: Kubernetes 1.X → 1.Y
- アップグレードパス: 直接アップグレード可能 or 段階的アップグレード必要

---

## 🚨 URGENT UPGRADE NOTES

**最優先事項**: リリースノートの "Urgent Upgrade Notes" セクションを確認し、以下を記載してください：

- 即座に対応が必要な変更
- クラスタの動作に影響する重大な変更
- データ損失や可用性に影響する問題

**記載形式**:
```
### [タイトル]
- **影響**: どのような影響があるか
- **対応**: 何をすべきか
- **期限**: いつまでに対応すべきか
```

もし該当項目がない場合は "該当なし" と明記してください。

## 🔒 Security Updates

セキュリティ関連の変更を記載：

### CVE修正

- **CVE-YYYY-XXXXX**: 概要と影響
- 修正内容
- 推奨アクション

### セキュリティ強化

- 追加されたセキュリティ機能
- デフォルト設定の変更

### [変更内容]

- **移行方法**: どう対応すべきか
- **影響度**: Critical/High/Medium/Low

影響があるマニフェストについて：

## 📋 Pre-Upgrade Checklist

アップグレード前に実施すべき項目：

- [ ] manifests/ 内の非推奨API使用状況を確認
- [ ] 影響を受けるマニフェストを修正
- [ ] 現在のクラスタ状態をバックアップ
- [ ] Urgent Upgrade Notesの対応完了を確認
- [ ] ステージング環境でテストアップグレード実施
- [ ] モニタリング・アラート設定の確認
- [ ] ロールバック手順の準備
- [ ] メンテナンスウィンドウの確保
- [ ] 関係者への通知

---

**分析の重要なポイント**:
1. リリースノートの事実に基づいて記載してください
2. 推測は避け、不明な点は「要確認」と記載してください
3. Urgent Upgrade Notes は必ず確認し、存在する場合は詳細に記載してください
4. manifests/ ディレクトリ内のすべてのファイルを確認し、具体的な影響を分析してください
5. 非推奨APIは具体的なファイル名と行番号を示してください
6. actionable な情報を提供することを最優先してください
